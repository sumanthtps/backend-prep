# Partitioning & Sharding
Partitioning and sharding help manage **large datasets efficiently** by distributing data across multiple tables or databases.  

---

## **1Ô∏è‚É£ What is Partitioning?**  
Partitioning **splits a large table into smaller, manageable pieces (partitions)** while still acting as a single table for queries.  

üöÄ **Benefits of Partitioning:**  
‚úî Faster queries by scanning only relevant partitions.  
‚úî Reduces index size for better performance.  
‚úî Speeds up `DELETE` & `UPDATE` by affecting fewer rows.  
‚úî Improves table maintenance (`VACUUM`, `ANALYZE`).  

---

## **2Ô∏è‚É£ Types of Partitioning in PostgreSQL**  

| Partitioning Type | How It Works | Best For |
|------------------|-------------|----------|
| **Range Partitioning** | Divides data based on **value ranges** (e.g., dates, numbers) | Time-series data, logs |
| **List Partitioning** | Divides data based on **specific values** (e.g., regions, categories) | Multi-region user data |
| **Hash Partitioning** | Uses a **hash function** to evenly distribute data | Large-scale workloads, even distribution |
| **Composite Partitioning** | Combination of **two partitioning types** | Complex datasets |

## **3Ô∏è‚É£ Range Partitioning Example**  
‚úÖ **Step 1: Create a Partitioned Table (Base Table)**  
```sql
CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    order_date DATE NOT NULL,
    amount NUMERIC
) PARTITION BY RANGE (order_date);
```

‚úÖ **Step 2: Create Partitions Based on Date Ranges**  
```sql
CREATE TABLE orders_2023 PARTITION OF orders 
FOR VALUES FROM ('2023-01-01') TO ('2023-12-31');

CREATE TABLE orders_2024 PARTITION OF orders 
FOR VALUES FROM ('2024-01-01') TO ('2024-12-31');
```
üöÄ **Now, queries like:**  
```sql
SELECT * FROM orders WHERE order_date = '2023-06-15';
```
Will automatically **scan only the `orders_2023` partition** instead of the entire table!

## **4Ô∏è‚É£ List Partitioning Example**  
‚úÖ **Partition Users by Region:**  
```sql
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    name TEXT,
    region TEXT
) PARTITION BY LIST (region);
```

‚úÖ **Creating Partitions for Specific Regions:**  
```sql
CREATE TABLE users_asia PARTITION OF users 
FOR VALUES IN ('India', 'China', 'Japan');

CREATE TABLE users_europe PARTITION OF users 
FOR VALUES IN ('Germany', 'France', 'UK');
```
üöÄ **Now, queries filter automatically:**  
```sql
SELECT * FROM users WHERE region = 'India'; -- Scans only users_asia
```

## **5Ô∏è‚É£ Hash Partitioning Example**  
Used when **data distribution must be even** (e.g., user IDs).  

‚úÖ **Step 1: Create a Hash-Partitioned Table**  
```sql
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    name TEXT
) PARTITION BY HASH (id);
```

‚úÖ **Step 2: Create Hash Partitions**  
```sql
CREATE TABLE users_part_0 PARTITION OF users 
FOR VALUES WITH (MODULUS 4, REMAINDER 0);

CREATE TABLE users_part_1 PARTITION OF users 
FOR VALUES WITH (MODULUS 4, REMAINDER 1);
```
üöÄ **Now, user IDs are evenly distributed across partitions!**

## **6Ô∏è‚É£ Performance Optimization with Partitioning**
‚úî **Create indexes on each partition**:  
```sql
CREATE INDEX idx_orders_2023 ON orders_2023(order_date);
```
‚úî **Enable Partition Pruning** (PostgreSQL auto-skips irrelevant partitions).  
‚úî **Use `EXPLAIN ANALYZE` to check partition usage**.

## **7Ô∏è‚É£ What is Sharding? (Distributed Partitioning)**
Sharding **distributes data across multiple databases/servers** instead of within a single table.  

üöÄ **How It Works:**  
‚úî Each shard **stores a subset of data** (e.g., users by country).  
‚úî Queries are routed **only to relevant shards**.  
‚úî Increases **scalability & fault tolerance**.  

‚úÖ **PostgreSQL Sharding Approaches:**  
1Ô∏è‚É£ **Foreign Data Wrappers (FDW)** ‚Äì Query multiple PostgreSQL instances like a single database.  
2Ô∏è‚É£ **Citus Extension** ‚Äì Makes PostgreSQL **horizontally scalable** by distributing queries across nodes.  
3Ô∏è‚É£ **Manual Application-Level Sharding** ‚Äì The application decides which database stores the data.
